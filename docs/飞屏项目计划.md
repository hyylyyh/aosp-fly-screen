# 跨屏拖拽（飞屏）项目计划

> 目标：在车载多屏场景下，实现应用从一个屏幕拖拽到另一个屏幕  
> 技术方向：AMS / WMS / Input 三者联动

---

## 一、项目概述

### 1.1 什么是飞屏

车载多屏场景（中控屏 + 副驾屏 + 后排屏）下，用户可以通过拖拽手势，把一个应用从当前屏幕"拖"到另一个屏幕上运行。

**示例场景**：
- 中控屏在放音乐
- 用户长按音乐应用标题栏，进入拖拽模式
- 手指滑到副驾屏，松手
- 音乐应用自动在副驾屏打开，中控屏回到桌面

### 1.2 完整版 vs 简化版

| 功能 | 完整版（真正迁移） | 简化版（重新启动） |
|---|---|---|
| 长按进入拖拽 | ✅ | ✅ |
| 镜像窗口跟随手指 | ✅ SurfaceControl | ✅ SurfaceControl |
| 松手后迁移应用 | ✅ Task 真正迁移到目标 Display | ⚠️ 在原屏幕 finish，目标屏幕重新 startActivity |
| 应用状态保留 | ✅ 完整保留（Task 迁移） | ⚠️ 重新启动，状态需应用自己恢复 |
| 预估时间 | 1-2 个月 | 2-3 周 |
| 难度 | 需要改 AMS 底层 Task 绑定逻辑 | 只需要 AMS 的 setLaunchDisplayId |

**当前策略**：先做**简化版**，面试时展示效果和架构理解。完整版作为后续深入研究方向。

## 二、架构设计（三层模型）

```
┌─────────────────────────────────────────┐
│  第一层：Input（SystemUI / 手势服务）      │
│  ─────────────────────────────────────  │
│  • 用 InputMonitor 监听系统级触摸事件      │
│  • 识别长按手势（>1000ms + 移动<20px）    │
│  • 进入拖拽模式，拦截后续 MOVE/UP 事件     │
│  • 手指移动时通知 WMS 更新镜像窗口位置      │
└──────────────┬──────────────────────────┘
               │ Binder 调用
┌──────────────▼──────────────────────────┐
│  第二层：WMS（窗口管理服务）               │
│  ─────────────────────────────────────  │
│  • 收到拖拽开始通知                        │
│  • 创建 SurfaceControl 镜像窗口            │
│  • 实时更新镜像窗口位置（跟随手指）         │
│  • 拖拽结束时销毁镜像窗口                  │
└──────────────┬──────────────────────────┘
               │ 松手后
┌──────────────▼──────────────────────────┐
│  第三层：AMS（Activity 管理服务）          │
│  ─────────────────────────────────────  │
│  • 判断松手时落在哪个 Display              │
│  • 原屏幕：finish 当前 Activity           │
│  • 目标屏幕：setLaunchDisplayId 重新启动   │
└─────────────────────────────────────────┘
```

**关键设计决策**：
- **WMS 不直接监听 Input 事件**！Input 监听在 SystemUI 层做，WMS 只负责窗口
- SystemUI 和 WMS 之间通过 Binder/AIDL 通信
- AMS 只做最后的"先关后开"，不做复杂的 Task 迁移

## 三、技术方案详解

### 3.1 Week 1：Input 层长按检测

**目标**：在 SystemUI 中实现长按手势检测，超过 1000ms 触发拖拽模式。

**核心类**：
- `LongPressGestureMonitor`（自定义）
- `InputMonitor`（系统 API）
- `InputEventReceiver`（系统 API）

**关键逻辑**：
```
ACTION_DOWN    → 记录按下的时间、坐标
ACTION_MOVE    → 计算移动距离，超过阈值取消长按
ACTION_UP      → 判断是否满足长按条件（时长>1000ms && 距离<20px）
                → 满足：进入拖拽模式，通知 WMS
                → 不满足：忽略
```

**重要**：`finishInputEvent(event, false)` — 不消费事件，正常应用仍能收到触摸。

**代码位置**：`frameworks/base/packages/SystemUI/`

---

### 3.2 Week 2：WMS 镜像窗口

**目标**：拖拽时显示一个半透明的应用镜像，跟随手指移动。

**核心类**：
- `SurfaceControl`（截图当前窗口内容）
- `WindowManager.addView()` / `WindowManager.updateViewLayout()`
- `DragDropController`（参考 SystemUI 现有的拖拽逻辑）

**关键逻辑**：
```
1. 收到拖拽开始通知
2. SurfaceControl.captureLayers(windowToken, bounds) 截图当前窗口
3. 创建 TYPE_DRAG 类型的浮动窗口
4. 手指移动时：wm.updateViewLayout(dragView, newParams)
5. 松手时：wm.removeView(dragView)
```

**代码位置**：`frameworks/base/services/core/java/com/android/server/wm/`

---

### 3.3 Week 3：AMS 跨屏启动

**目标**：松手后，判断目标屏幕，重新启动应用到目标 Display。

**核心 API**：
- `ActivityOptions.setLaunchDisplayId(displayId)`
- `Activity.finish()`
- `context.startActivity(intent, options.toBundle())`

**关键逻辑**：
```java
// 1. 判断松手时落在哪个 Display
Display targetDisplay = getDisplayAt(x, y);
int targetDisplayId = targetDisplay.getDisplayId();

// 2. 在原屏幕 finish
activity.finish();

// 3. 在目标屏幕重新启动
Intent intent = new Intent();
intent.setComponent(activity.getComponentName());
ActivityOptions options = ActivityOptions.makeBasic();
options.setLaunchDisplayId(targetDisplayId);
context.startActivity(intent, options.toBundle());
```

**代码位置**：`frameworks/base/services/core/java/com/android/server/am/`

## 四、深度知识点（面试用）

### 4.1 InputMonitor 底层链路

```
Java: InputManager.monitorInput()
    ↓ JNI
Native: InputManagerService::monitorInput()
    ↓
Native: InputDispatcher::monitorInput()
    ↓
Native: 创建 socket pair → InputChannel(server + client)
    ↓
Native: server 注册到 InputDispatcher.mMonitoringChannels
    ↓
Java: 返回 client 的 InputChannel
    ↓
Java: 创建 InputEventReceiver 监听
```

**面试话术**：
> "我用 InputMonitor 做系统级手势监听，但这不是简单调 API。底层看，InputDispatcher 会创建一对 InputChannel，server 端注册到 mMonitoringChannels 列表里。当触摸事件从 EventHub 读到后，InputDispatcher.dispatchMotionLocked() 里会先把事件发给所有 monitor，再发给焦点窗口。所以 monitor 比焦点窗口优先收到事件。"

### 4.2 WMS 与 InputDispatcher 的关系

- WMS **不直接消费**输入事件
- WMS 通过 `InputWindowHandle` 告诉 InputDispatcher 每个窗口的输入区域
- InputDispatcher 根据 Focused Window 决定事件发给谁
- WMS 实现了 `InputManagerService.WindowManagerCallbacks` 接口，但只处理窗口状态变化（如 InputChannel 断开、ANR），不处理触摸事件本身

### 4.3 AMS 跨 Display 启动

- `ActivityOptions.setLaunchDisplayId()` 从 Android 8.0 引入
- 底层通过 `LaunchParamsController` 将 Task 分配到指定 Display 的 TaskDisplayArea
- 真正的 Task 跨 Display 迁移需要改 `Task.reparent()` 和 `RootWindowContainer`，涉及 `DisplayContent` 切换
